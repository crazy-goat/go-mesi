# Changelog

## [0.13.0] - Unreleased

### Added
- CLI `-shared-http-client` flag: when set, creates a shared `http.Client` with connection pooling and SSRF-safe transport for all ESI includes within a single invocation. Reduces latency for pages with many includes to the same origin (#227)
- CLI `-cache-key-template` flag: custom cache key template with `${url}` placeholder substitution. Example: `-cache-key-template='myapp:${url}'` (#222)

### Fixed
- Apache README: remove incorrect "Mutex around Parse()" claim for MPM Worker/Event. The mutex was never implemented; `dlopen`/`dlsym` already runs in `child_init` (before threads start), and libgomesi is built with Go's goroutine-safe runtime. The MPM Compatibility table now accurately describes the actual thread-safety model (#94).

### Added
- Traefik memory cache backend: `cacheBackend: memory`, `cacheSize`, and `cacheTTL` config options wire the in-memory LRU cache into Traefik ESI processing. Duplicate `<esi:include>` URLs within TTL are served from cache, reducing backend load (#234)
- libgomesi `InitCache` and `FreeCache` exports enable shared cache across C-based consumers (nginx, Apache, PHP extension). Supported backend: `"memory"` with configurable size and TTL (#232)
- nginx cache backend directives: `mesi_cache_backend memory`, `mesi_cache_size`, and `mesi_cache_ttl` wire the in-memory LRU cache into nginx ESI processing. Duplicate `<esi:include>` URLs within TTL are served from cache, reducing backend load (#232)
- Apache memory cache backend directives: `MesiCacheBackend memory`, `MesiCacheSize`, and `MesiCacheTTL` wire the in-memory LRU cache into Apache mESI output filtering. The directives validate strictly (`MesiCacheBackend` must be `memory` or empty — any other value is rejected at startup to prevent silent cache disablement due to typos; `MesiCacheSize` in `[1, 1000000]`; `MesiCacheTTL` in `[0, 86400]` seconds). A duplicate-include fixture and backend hit-count assertion in `test.sh` prove the cache dedups within a single response (#174)
- CLI memory cache backend: `-cache-backend=memory`, `-cache-size`, and `-cache-ttl` flags wire the `mesi.MemoryCache` into CLI invocations so duplicate `<esi:include>` URLs are served from cache within a single run (#207)
- CLI Redis cache backend: `-cache-backend=redis`, `-cache-redis-addr`, `-cache-redis-password`, and `-cache-redis-db` flags wire the `cache_redis.RedisCache` into CLI invocations, enabling Redis-backed ESI caching from the command line (#212)
- CLI Memcached cache backend: `-cache-backend=memcached` and `-cache-memcached-servers` flags wire the `cache_memcached.MemcachedCache` into CLI invocations, enabling Memcached-backed ESI caching from the command line (#217)
- RoadRunner memory cache backend: `cache_backend: memory`, `cache_size`, and `cache_ttl` config options wire the in-memory LRU cache into the RoadRunner middleware. Duplicate `<esi:include>` URLs within TTL are served from cache, reducing backend load. Per-worker-pool scope (each RoadRunner worker goroutine owns its own `MemoryCache`) is documented in the README and the new `examples/roadrunner-memory-cache.yaml` fixture (#236)
- RoadRunner Memcached cache backend: `cache_backend: memcached` and `cache_memcached_servers` config option wire the `cache_memcached.MemcachedCache` into the RoadRunner middleware, enabling Memcached-backed ESI caching (#245)
- libgomesi `InitCacheWithConfig` C entrypoint: takes a JSON config blob alongside the existing `backend`/`size`/`ttl` params, enabling `redis` (addr/password/db) and `memcached` (server list) configuration for C consumers like Apache. Calling on the `redis`/`memcached` backends without proper config returns `-1` so a typo never silently falls back to no cache (#175)
- PHP extension `cache_backend = "redis"` for `parse_with_config()`: takes `cache_redis_addr` (required `"host:port"` with port in `[1, 65535]`), optional `cache_redis_password`, and optional `cache_redis_db` (in `[0, 15]`). The extension renders a JSON config blob (mirroring `libgomesi`'s `redisConfig` struct) and routes through `InitCacheWithConfig`. Strict validation rejects mismatched backends (e.g. `cache_redis_addr` with `cache_backend = "memory"`), out-of-range ports, malformed `host:port` tokens, non-string passwords, and passwords containing control chars / `"` / `\` so the rendered JSON cannot be broken by hostile operator input. The legacy `parse_with_config()` in-process state is extended to remember the rendered `configJSON` per backend, so repeated calls with identical Redis configuration skip `InitCacheWithConfig` and never replace `sharedCache` with a fresh, empty instance (#231)
- Apache Redis cache backend directives: `MesiCacheBackend redis`, `MesiCacheRedisAddr` (host:port), `MesiCacheRedisPassword`, and `MesiCacheRedisDB` (0..15) wire the `cache_redis.RedisCache` into Apache mESI output filtering, enabling Redis-backed ESI caching shared across Apache workers and instances. Strict validation: `MesiCacheBackend` accepts only `memory`, `redis`, `memcached`, or empty; `MesiCacheRedisAddr` requires a colon and a 1..65535 port (rejects whitespace, control chars, and JSON-meta chars so the rendered JSON is safe); `MesiCacheRedisPassword` rejects embedded control chars with a generic error that does not leak the value into error logs; `MesiCacheRedisDB` requires a non-negative integer in `[0, 15]`. Malformed config returns `-1` from libgomesi and is logged; ESI continues without cache (#175)
- Apache Memcached cache backend directive: `MesiCacheBackend memcached` and the new `MesiCacheMemcachedServers` directive (space-separated `host:port` list, max 64 entries) wire the `cache_memcached.MemcachedCache` into Apache mESI output filtering, enabling shared Memcached-backed ESI caching. Strict validation: each entry must contain a colon and a 1..65535 port (rejects whitespace, control chars, JSON-meta chars, decimal/alpha ports, empty values, and >64 entries); an empty server list with the memcached backend produces a deterministic libgomesi `servers required` error rather than a silent `localhost:11211` default. Malformed config returns `-1` from libgomesi and is logged; ESI continues without cache (#176)
- nginx Memcached cache backend directive: `mesi_cache_backend memcached` and the new `mesi_cache_memcached_servers` directive (space-separated `host:port` list) wire the `cache_memcached.MemcachedCache` into nginx ESI processing via `InitCacheWithConfig`. An empty server list with the memcached backend produces a deterministic libgomesi `servers required` error. Init failures are logged at ERR level; ESI continues without cache (#240)
- nginx Redis cache backend directives: `mesi_cache_backend redis`, `mesi_cache_redis_addr` (host:port), `mesi_cache_redis_password`, and `mesi_cache_redis_db` (0..15) wire the `cache_redis.RedisCache` into nginx ESI processing via `InitCacheWithConfig`, enabling Redis-backed ESI caching shared across nginx worker processes and instances. Defaults: addr `localhost:6379`, no password, DB 0. Init failures are logged at ERR level; ESI continues without cache (#237)
- PHP extension `parse_with_config()` plus the in-memory cache backend: takes an associative `config` array alongside the existing `parse()` arguments and supports `cache_backend` (only `"memory"` is exposed — any other value triggers an `E_WARNING` rather than silently caching nothing), `cache_size` (integer in `[1, 1_000_000]`, rejection-warning outside this range), and `cache_ttl` (integer in `[0, 86_400]` seconds, rejection-warning outside this range). The extension tracks the last completed `InitCache` configuration in process-local state so repeated calls with the same parameters don't wipe the in-memory cache between requests — a fresh `InitCache` build always starts empty, so without this guard every cache hit would silently regress to a miss. A fork-based PHPT asserts the cache dedups `<esi:include>` URLs in-process; a second PHPT asserts every validation path (#226)
- Proxy cache backend CLI flags: `--cache-backend` (memory/redis/memcached), `--cache-size`, `--cache-ttl`, `--cache-redis-addr`, `--cache-redis-password`, `--cache-redis-db`, and `--cache-memcached-servers` wire all three cache backends into the standalone proxy server. Unknown backend values are rejected with a clear error — never silently disables cache (#331)
- nginx SSRF protection directive: `mesi_block_private_ips on | off` controls whether ESI includes to private/reserved IP addresses (loopback, RFC 1918, CGNAT, link-local, cloud metadata `169.254.169.254`, benchmark/documentation ranges) are blocked at dial time. The nginx module now calls `libgomesi.ParseWithConfig` (instead of `Parse`) so the `BlockPrivateIPs` flag is honoured. Default is `on` (#193)
- Traefik `blockPrivateIPs` plugin config option: controls whether ESI includes to private/reserved IP addresses are blocked. It is wired through to `mesi.EsiParserConfig.BlockPrivateIPs` and defaults to `true`, preserving the previous hardcoded behaviour. Under Yaegi the dial-time IP-blocking transport is stubbed (see `servers/traefik/Dockerfile`), so the option is configurable but the actual dial-time enforcement is not active in the interpreted plugin — URL-level `allowedHosts` protection (#200) is the effective control there (#195)
- PHP extension `block_private_ips` option for `parse_with_config()`: controls whether ESI includes to private/reserved IP addresses (loopback, RFC 1918, CGNAT, link-local, cloud metadata `169.254.169.254`, benchmark/documentation ranges) are blocked at dial time, closing the SSRF gap in the PHP extension. The option defaults to `true` (secure by default) — the extension now routes through `libgomesi.ParseWithConfig` (instead of `Parse`) and (re)builds the shared HTTP client's SSRF-safe transport only when the requested value changes between calls. A non-boolean value is rejected with `E_WARNING` and `parse_with_config()` returns `false`. The legacy `\mesi\parse()` entrypoint is unchanged in signature and keeps the module-init behaviour (no blocking) until a `parse_with_config()` call opts in (#185)

### Changed
- **`<esi:include fetch-mode="ab" ab-ratio="…">` now rejects malformed input.** Previously the parser silently substituted the documented `{A:50, B:50}` default for every malformed value (missing colon, extra colons, negative numbers, decimals, non-integer operands, oversized integers, both-sides-zero). It now returns `*ErrInvalidABRatio` through the existing include-error path, surfacing the operator's actual input in logs and the `IncludeErrorMarker` (#315).
- **`<esi:include max-depth="…">` now rejects malformed input and clamps to a documented upper bound.** Previously the parser silently substituted the parent's `MaxDepth` for every malformed value (non-numeric, negative, decimal, oversize — including values past `MaxUint64`), and silently accepted any parsed-but-unbounded value via `uint(v)+1`. It now returns `*ErrInvalidMaxDepth` through the configured logger (visible via `Debug` and any logger implementing `Warn`) and preserves the parent's `MaxDepth` for the rejected override. Empty / whitespace-only attributes continue to be treated as "no override" so existing templates that omit the attribute or pass a blank one are not silently downgraded. Accepted values are bounded by `MaxMaxDepth` (10,000) so the `+1` arithmetic that clamps the parent's depth can never approach any platform's `uint` wrap boundary (#317).
- **BREAKING CHANGE: nginx `mesi_block_private_ips` now defaults to `on`.** Previously the nginx module called `libgomesi.Parse` which never set `BlockPrivateIPs`, so every `<esi:include>` to a private/reserved IP (loopback, RFC 1918, cloud metadata `169.254.169.254`, etc.) succeeded with no protection. The module now calls `ParseWithConfig` and the new `mesi_block_private_ips` directive defaults to `on`. Deployments that intentionally include from internal/private backends must explicitly set `mesi_block_private_ips off;` (#193).

### Fixed
- `MESIParse` no longer silently normalizes a negative `MaxConcurrentRequests` to zero. Invalid values are surfaced through the configured logger via `Debug` (or `Warn` for loggers implementing `LoggerWarn`) before normalizing, matching the pattern established by the `max-depth` (#317) and `ab-ratio` (#315) validation fixes (#329).
- Race condition in concurrent ESI include fetches: when two parallel workers process identical `<esi:include>` URLs both miss the cache and both perform upstream round-trips, returning different responses. A per-key inflight lock now serialises cache-miss fetches for the same URL so the second worker blocks until the first populates the cache and then serves from it — fixing the same-page cache-dedup guarantee for both in-memory and external cache backends (#329).
- Nginx module: `cache_initialized` now tracks the last-configured backend. Switching between `mesi_cache_backend memory` and `mesi_cache_backend memcached` (different `location` blocks in the same server) re-initialises the shared cache via `InitCacheWithConfig` instead of silently keeping the first-initialised backend forever (#329).
- `selectUrl` integer-sum overflow at high ratios guarded against `math.MaxInt` saturation (#315).
- `OverrideConfig` `max-depth` path previously accepted any parseable value and propagated it through the `uint(v)+1` clamp with no upper bound. Inputs above `MaxMaxDepth` (10,000) or syntactically invalid now produce `*ErrInvalidMaxDepth` so operator typos and hostile templates both surface through the logger while the parent's `MaxDepth` survives untouched — preventing a single misconfigured include from silently disabling all nested ESI processing under it (#317).
- `mesi` test suite: removed a hand-rolled `contains` / `containsHelper` substring-search reimplementation from `fetch_test.go` and `ssrf_test.go`; the same call sites now use the standard-library `strings.Contains`. A new `TestNoDuplicatedHelper` regression test fails if any local substring helper is reintroduced (#117).
- `servers/apache/Dockerfile` now verifies the Go tarball SHA256 checksum before extraction, replaces the hardcoded `go1.23.0.linux-amd64` download with parameterised `GO_VERSION` and `GO_SHA256` build args, and uses `curl -fsSL` instead of `wget` for better failure handling. The `TARGETARCH` build arg is wired into the download URL so the image can be built for non-amd64 platforms via `--platform` (#101).
- `servers/apache/build.sh` no longer masks install failures. The script previously ran `cp "$LIBGOMESI_SO" /usr/lib/libgomesi.so 2>/dev/null || sudo cp "$LIBGOMESI_SO" /usr/lib/libgomesi.so`, silently swallowing every class of error (permission denied, missing source, read-only destination, disk full, ENOSPC) and falling back to a possibly-successful `sudo install` against an unrelated path. The rewritten script enables `set -euo pipefail`, tests install-prefix writability via `[[ -w "$INSTALL_PREFIX" ]]` before invoking `install -m 0644`, names the destination directory in the sudo-prompt message so the operator knows what password unlocks, allows overriding the install prefix with `INSTALL_PREFIX` (defaults to `/usr/lib`; convenient for `/usr/local/lib` layouts), records explicitly when the source `libgomesi.so` is missing or the `go` toolchain isn't on PATH, and emits a clear error when `apxs` / `apxs2` are not installed. A new `servers/apache/test_build_sh.sh` exercises six deterministic scenarios (non-root success, custom prefix, non-existent prefix, missing apxs, missing source + missing go, stderr preserved); each uses a stub `sudo` so the sudo branch is testable regardless of host euid (#102).
